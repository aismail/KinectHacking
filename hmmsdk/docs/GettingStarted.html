HmmSDK Development: Getting Started

If you would like to contribute to the HmmSDK project, you will need to talk to one of the admins of the project first. The main component of HmmSDK is a library program written in Java, but you don't have to be a Java programmer to participate in this project. Any contribution commensurate with the overall goal of the project will be appreciated. Please visit its home page for more information.

Once you decide that you would participate in the project, please follow the instructions below. This is essentially a summary of many relevant documents found in SourceForge.net Doc Page.

Setting Up Your Development Environment

  1. If you don't have an account at SourceForge.net, then the first thing you need to do is to register as a user.
  2. Send email to the project admins with your intention to join. Once you are approved, you will be given the Developer Privilege and/or any other appropriate permissions depending on the role you would like to assume in the project.
  3. Join the developer mailing list. It can take up to a day before your request for subscription is processed. So, it's important to do this as soon as possible. (This is the time when you will need most help from fellow developers.)
  4. You will need SSH clients (compliant with either SSH1 or SSH2 protocol) in order to be able to access CVS repository. I use PuTTY on Windows platform. Please refer to this document on SourceForge.net for more information.
  5. Create a CVS home directory. This is a one-time process before you can use CVS on SourceForge.net. Here's detailed instructions.
  6. Generate SSH key pairs. Here's the link to the SourceForge document. You can follow the steps in this note if you are using PuTTY on Windows.
  7. Upload the public key. This step is also very well explained in the SSH document or in this section of the PuTTY document.
  8. Install/configure CVS client. If you are using Windows, WinCVS is recommended. Set your CVSROOT to yourusername@cvs.sourceforge.net:/cvsroot/hmmsdk with the ssh authentication protocol. This document has a very nice step-by-step instruction on how to set up WinCVS and PuTTY together. If you are new to CVS, then you might want to spend some time familiarizing yourself with basic functionalities of CVS. Again documentation on SourceForge.net is a good starting point.
  9. Now the hard part is over. It is a good time to subscribe to the CVS commit message mailing list in order to get informed of any changes made to the files in the repository.
  10. For a list of active modules in the project, try 'cvs co -c' or use any correponding command in your CVS client. Check out necessary modules and you are all set! (Please ask for help from existing members regarding which modules you need to check out.) If you are a software developer, you will probably want to check out the HmmLib module first (among others). The following steps are only necessary if you intend to do some programming. (Obviously, you will need to have JDK (v1.4 or later) and Ant installed on your system before you can proceed. We will assume that it has already been done.)
  11. Once you check out the HmmLib module, please spend a bit of time browsing through the directory structure. (Most modules have a very similar structure.) The build files are kept under the directory, bin. Build.xml is the common Ant build script, and some user-specific properties can be set in the build.properties file (not being used as of this writing).
  12. Some of the libraries in the lib directory are needed for Ant build process (custom ant tasks). You can either include these lib files (bluecraft.common-M.m.jar and bluecraft.tools-M.m.jar) in your system-level classpath, or you can use some kind of shell scripts to set the classpath before you call Ant. Please look at the Windows build.bat file for an example. (Note that Ant should be invoked from the bin directory.)
  13. You can find Java source files under src/java. Please try building the hmmlib module at this point. You might have to manually create a directory called build as a sibling of bin before you can build the module. (Use ant -projecthelp to list available targets. Compileall is the default target.) If everything goes well, then the output should be generated under the build directory.
  14. BTW, do not check in the file, com.bluecraft.hmmlib.buildnum.txt, under the bin directory. This file is used to generate version/build number for official releases.
  15. If you would like to build all modules whose names start with hmm, then check out all modules first and use the master Ant build file under bin in the HmmAll module. Include definitions like hmmlib.project.dir=E:/hmmsdk/hmmlib and hmmbox.project.dir=E:/hmmsdk/hmmbox in your build.properties file.

Getting Support

As with many open-source projects, it's very important for us to help each other out. If you are having any trouble setting up your initial environment, please send email to the developer mailing list. One of us will gladly help.
